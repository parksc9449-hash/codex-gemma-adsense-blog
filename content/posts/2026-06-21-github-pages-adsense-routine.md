# 깃허브 페이지 블로그를 오래 운영하는 현실적인 루틴

## 서버보다 중요한 것은 반복 가능한 과정

GitHub Pages는 정적 파일을 배포하기 때문에 서버 비용을 거의 생각하지 않아도 됩니다. 하지만 블로그 운영에서 더 자주 무너지는 부분은 인프라가 아니라 발행 루틴입니다. 매일 새 글을 만들 수 있더라도, 제목이 겹치고 본문이 얕고 정책 페이지가 비어 있으면 승인과 검색 노출 모두에서 불리해집니다.

운영 루틴은 글감을 찾는 일, 초안을 만드는 일, 검수하는 일, 배포 후 확인하는 일을 분리해두는 방식으로 설계하는 편이 좋습니다. 한 번에 모든 것을 끝내려 하면 누락이 생기기 쉽고, 특히 정적 블로그는 배포 결과물이 실제 URL에서 잘 보이는지 확인하는 과정이 중요합니다.

## 기본 구조

| 영역 | 권장 방식 | 이유 |
| --- | --- | --- |
| 초안 작성 | 로컬 Gemma 4 | 반복 초안의 토큰 비용을 줄입니다. |
| 코드 수정 | Codex OAuth | 저장소 구조 변경과 빌드 오류 해결에 적합합니다. |
| 배포 | GitHub Actions | 정적 파일을 자동으로 Pages에 올립니다. |
| 검수 | 승인 감사 스크립트 | 필수 페이지와 중복을 빠르게 확인합니다. |

## 발행 전 확인할 것

- 같은 주제의 기존 글이 있는지 확인합니다.
- 개인정보처리방침, 소개, 문의 페이지가 실제로 연결되어 있는지 확인합니다.
- 광고 코드는 승인 전까지 비활성 상태로 둡니다.
- 제목이 과장된 수익 약속처럼 보이지 않는지 점검합니다.
- 본문에 표, 체크리스트, 실제 설정 경로처럼 독자가 바로 확인할 수 있는 정보가 있는지 봅니다.
- 검색 등록 후 제출할 `sitemap.xml`과 `robots.txt`가 최신 글을 포함하는지 확인합니다.

## 주간 운영 루틴 예시

| 요일 | 작업 | 확인 기준 |
| --- | --- | --- |
| 월요일 | 주제 후보 정리 | 기존 글과 겹치지 않는가? |
| 화요일 | 로컬 AI 초안 생성 | 제목, 설명, 소제목이 있는가? |
| 수요일 | 사람 편집 | 과장 표현과 사실 오류를 줄였는가? |
| 목요일 | 빌드와 승인 감사 | `npm run build`, `npm run approve`가 통과하는가? |
| 금요일 | 검색 도구 확인 | 사이트맵 제출 상태와 색인 오류를 확인했는가? |

이 표는 엄격한 일정표라기보다 누락을 줄이기 위한 기준입니다. 글을 많이 쓰는 주간에는 초안 생성보다 검수 시간을 더 확보하고, 검색 도구에서 오류가 보이는 주간에는 새 글보다 기존 글 정리에 집중하는 편이 낫습니다.

## 자동화의 적정선

자동화는 초안과 반복 작업을 줄이는 데 쓰는 편이 안전합니다. 검색 노출을 노린 대량 복제, 출처 없는 수치, 광고 클릭 유도 문구는 장기 운영에 도움이 되지 않습니다. 특히 AdSense 승인 전에는 사이트의 신뢰 신호가 중요하므로, 글 수보다 기본 페이지와 편집 품질을 우선하는 편이 낫습니다.

로컬 Gemma 4는 반복 초안 비용을 줄이는 데 유용하지만, 최종 판단까지 맡기기에는 부족한 부분이 있습니다. 수치, 정책, 서비스 화면처럼 바뀔 수 있는 내용은 직접 확인하고, 글 말미에는 독자가 실행할 수 있는 점검 항목을 남겨야 합니다. Codex OAuth는 저장소 구조와 자동화 스크립트를 다루는 데 활용하고, 콘텐츠 판단은 감사 스크립트와 사람 검수를 함께 쓰는 식으로 역할을 나누면 운영 부담이 줄어듭니다.

## 배포 후 확인 루틴

배포가 끝나면 GitHub Actions가 초록색으로 끝났는지만 보지 말고 실제 공개 URL을 열어야 합니다. 홈 화면에 새 글이 노출되는지, 글 상세 페이지의 제목과 설명이 올바른지, 사이트맵에 새 URL이 들어갔는지 차례로 확인합니다.

검색 등록을 마친 뒤에는 Google Search Console과 Naver Search Advisor에서 같은 사이트맵 주소를 제출하고, 며칠 간격으로 색인 상태를 확인합니다. 색인이 바로 반영되지 않는 것은 자연스러운 일이므로, 오류가 없다면 새 글 발행과 기존 글 보강을 병행하는 편이 현실적입니다.

## 다음 단계

이 프로젝트에서는 `npm run generate`로 초안을 만들고, `npm run approve`로 승인 준비 상태를 확인한 뒤, `npm run build`로 정적 사이트를 생성합니다. 로컬 대시보드에서는 같은 작업을 버튼으로 실행할 수 있습니다.
